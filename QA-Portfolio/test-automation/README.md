## Test Automation
반복되는 테스트케이스는 자동화 스크립트를 작성하여 실행한다.

핵심은 모든 케이스를 자동화하지 않는다. 자동화 후보 기준은 보통 회귀 테스트로 자주 돌아야하거나
수동으로 하기에는 데이터 조합이 많거나 UI가 자주 안바뀌어 유지보수 비용이 낮은 것 중 선정한다.

반대로 탐색적 테스트, UX 검증, 자주 바뀌는 화면은 수동으로 남겨두는게 효율적이다.

<br/>

### 테스트 설계의 일반적인 흐름
#### 요구사항(Requirements)
기능 및 스펙 정의

#### 요구사항 분석
모호한 부분(누락된 부분)
엣지케이스
리스크 식별

#### 테스트 케이스 설계
정상/예외/경계값 등 전체 케이스 작성 (수동 포함)

#### 자동화 대상 선정
반복성이 높고 회귀리스크 크고 안정적인 케이스만 골라서 자동화

<br/>

### 자동화 설계
대부분의 UI 자동화는 **POM(Page Object Model)** 구조를 기본으로 한다.<br/>
페이지 단위로 셀렉터와 동작을 캡슐화하고, 테스트는 POM 메서드를 호출해 시나리오를 조립한다.

#### 설계 흐름
1. **자동화 대상 시나리오 확정** — TC 표에서 자동화 후보로 고른 케이스 정리
2. **페이지 / 컴포넌트 식별** — 시나리오를 화면 단위로 분해 (Login, Cart, Checkout 등)
3. **로케이터 전략 결정** — `data-testid` 우선 → role → text 순으로 폴백
4. **페이지 오브젝트 작성** — 페이지별 비즈니스 메서드와 상태 노출 메서드 정의
5. **테스트 데이터 분리** — 계정 / 상품 / 쿠폰 등은 fixture 로 분리
6. **테스트 케이스 작성** — POM 호출로 시나리오 조립, 어서션은 테스트에만 둔다
7. **공통 픽스처 / 훅** — 로그인 상태, beforeEach, 네트워크 mock 등
8. **실행 / CI** — 리트라이, 트레이스, 병렬화, 리포터 설정

<br/>

#### Top-down vs Bottom-up
- **Top-down**: 스펙 → POM 설계 → 테스트 작성. 큰 그림은 잡기 쉽지만 안 쓰는 메서드까지 만들기 쉽다.
- **Bottom-up**: 테스트부터 평문으로 작성 → 중복을 POM 으로 추출. 실제 필요만큼만 추상화된다.

학습 단계에서는 **Bottom-up** 을 권장한다.<br/>
처음부터 완벽한 POM 을 그리려 하면 추상화 수준이 과해지기 쉽다.

<br/>

#### POM 작성 규칙
- POM 안에서는 `expect` (어서션) 금지. 어서션은 테스트에서만 한다.
- 메서드 이름은 **비즈니스 언어** 로. `addToCart()` ⭕ / `clickAddBtn()` ❌
- 페이지 이동이 발생하는 메서드는 다음 페이지의 POM 인스턴스를 return 한다.
- 로케이터는 생성자에서 한 번만 정의하고 메서드에서 재사용한다.
- 한 POM 은 하나의 페이지 / 영역에 한정한다. 너무 커지면 컴포넌트 단위로 쪼갠다.

<br/>

#### 셀렉터 우선순위
1. `data-testid` — 테스트 전용 속성, 디자인 변경에 영향 없음
2. 역할 기반 (`getByRole`) — 접근성 트리 기반, 의미가 명확
3. 텍스트 (`getByText`) — 문구가 변하지 않는 경우만
4. CSS / XPath — 위 셋이 불가능할 때만 최후의 수단

<br/>

#### 안티 패턴
- `sleep(1000)` 같은 고정 대기 — 항상 조건 기반 대기(`waitFor`) 로
- 테스트 간 상태 공유 — 각 테스트는 독립적으로 실행 가능해야 한다
- 한 테스트에서 여러 시나리오 검증 — 한 테스트는 한 시나리오, 한 어서션 묶음
- POM 안에 분기 로직 — POM 은 단순 동작 노출, 흐름 분기는 테스트에서
