# Assignment · QA 입사 과제 대비 문제집
QA 엔지니어 입사 전형(서류·과제·테스트) 대비용 자작 문제 모음.
실제 입사 과제의 형식을 흉내내어 카테고리 / 난이도 / 평가 기준을 정해두었다.

## 카테고리
| 코드 | 카테고리 | 경로 | 설명 |
|---|---|---|---|
| R | 요구사항 분석 | [`requirements/`](requirements/README.md) | 주어진 요구사항을 5가지 기준으로 분해 |
| T | 테스트 케이스 작성 | [`testcases/`](testcases/README.md) | 기능 명세로부터 TC 표 도출 (동등 분할 / 경계값) |
| B | 버그 리포트 작성 | [`bugreports/`](bugreports/README.md) | 결함 상황을 표준 버그 리포트로 정리 |
| P | 시나리오 우선순위 | [`scenarios/`](scenarios/README.md) | 출시 전 검증 시나리오와 우선순위 산정 |
| A | 자동화 실기 | [`automation/`](automation/README.md) | Playwright / Selenium 자동화 코드 작성 |

## 진행 방법
1. 카테고리 폴더의 `README.md` 에서 양식과 문제 목록 확인
2. 각 문제 폴더의 `README.md` 에서 요구사항을 읽고, 권장 시간 내에 답안을 `---` 라인 아래 답안 작성란에 채움
3. 작성 후 동일 카테고리의 다른 문제 또는 AI 솔루션과 **비교 학습**
4. 회사 지원 시 해당 회사 도메인 문제를 반복 풀이

## 작성 원칙 (모든 카테고리 공통)
- 요구사항에 명시되지 않은 시스템·정책을 과하게 상상하지 말 것
- 시간 / 금액 / 수량 / 비율 같은 수치 임계값은 빠짐없이 항목화할 것
- 분기(App/Web, 로그인/비로그인, 사용자 등급, 자동/수동)는 별도 항목으로 분리할 것
- 본인 추측은 `(추측)` 표식으로 사실과 분리할 것

## 평가 기준 (카테고리 공통 골격)
| 기준 | 가중치 | 평가 내용 |
|---|---|---|
| 완전성 | 25% | 양식 항목 모두 작성, 누락 없음 |
| 정확성 | 25% | 요구사항을 임의로 해석·추가하지 않음 |
| 분류력 | 25% | 정책 / 예외 / 외부 시스템 / 우선순위를 올바르게 구분 |
| 구체성 | 25% | 수치·상태·역할·기대 결과가 모호하지 않음 |

> 카테고리별 세부 가중치는 각 카테고리 README 참조
