# QA 과제 모범답안 — 커머스 쿠폰 적용 기능

> "정답"은 없지만, 평가에서 **통과 수준 이상**으로 읽히는 산출물 예시.
> 핵심은 ① 가정을 명시하고 ② 경계·예외를 빠짐없이 도출하며 ③ 명세 빈틈을 리포트하는 것.

---

## 1. 요구사항 분석

### 1-1. 기능 요약 (정상 흐름)
- **액터:** 구매자(로그인 사용자)
- **핵심 기능:** 결제 단계에서 보유 쿠폰(정액/정률) 중 1개를 선택해 상품금액 합계에 할인을 적용하고, 다시 해제할 수 있다. 적용/해제 시 결제 금액이 즉시 갱신된다.
- **적용 제약:** 유효기간 내 + 최소 주문금액 충족 시에만 적용 가능. 적용 불가 쿠폰은 비활성 표시.
- **외부 시스템:** PG사 결제 (할인 반영된 최종 금액으로 결제 요청)

### 1-2. 모호하거나 누락된 부분 (식별)
명세 v1.0을 검토해 **정의되지 않았거나 해석이 갈리는 지점**을 식별했다. (각 항목은 4. 스펙 리뷰에서 정식 질문으로 전달)

| ID | 식별한 모호점 / 누락 | 왜 문제인가 |
|----|---------------------|-------------|
| Q1 | 할인 적용 기준 금액에 **배송비 포함 여부** 미정의 | 정률 쿠폰에서 배송비 포함 여부에 따라 할인액이 달라짐 |
| Q2 | 정률 할인의 **소수점 처리(절사/반올림/단위)** 미정의 | 구현마다 결과가 달라져 정산 불일치 발생 가능 |
| Q3 | **최소주문금액 판정 기준**(적용 전/후, 배송비 포함) 미정의 | 적용 가능 여부 판정이 흔들림 |
| Q4 | **할인액 > 상품금액**일 때 동작(0원/음수 방지) 미정의 | 결제금액 음수 가능성 |
| Q5 | **쿠폰 적용 후 장바구니 변경** 시 동작 미정의 | 적용 후 최소금액 미달 상태 처리 불명확 |
| Q6 | **쿠폰 중복 사용**(쿠폰+쿠폰, 쿠폰+포인트) 가능 여부 미정의 | "하나를 선택"이 쿠폰 1개 한정인지 불명확 |
| Q7 | **유효기간 경계**(시·분·초, 타임존) 미정의 | 만료 시점 판정 불명확 |
| Q8 | **결제 실패 시 쿠폰 상태 롤백** 정책 미정의 | 쿠폰 증발 또는 중복 사용 위험 |
| Q9 | **특정 상품/카테고리 한정 쿠폰** 명세 부재 | 전체 할인만 존재, 확장 시 재설계 위험 |

- 쿠폰 X: 정액 5,000원 할인 / 최소주문 30,000원

- 쿠폰 Z: 정액 5,000원 할인 / 최소주문 4,000원 (0원 방지 검증용)

## 3. 리스크 분석 (Top 3)

### R1. 금액 계산 정확성 — *결함 임팩트 최상*
정률 할인의 **소수점 처리(절사/반올림/단위)**, **최대 할인한도 적용**, **0원/음수 방지**는 한 줄만 틀려도 곧바로 **금전·정산 사고**로 직결된다. 사용자에게 과다 할인되면 회사 손실, 과소 할인되면 CS·신뢰 문제다. 정률 쿠폰은 입력 금액에 따라 결과가 연속적으로 변하므로 경계값(한도 도달·초과, 소수점) 검증이 필수다. → TC-B03~B05, B08

### R2. 쿠폰 사용 처리의 무결성 (중복 사용·롤백·동시성)
쿠폰은 **1회성 자산**이다. 결제 실패 시 롤백이 누락되면 사용자는 쓰지도 못한 쿠폰을 잃고(CS 폭증), 반대로 동시성 제어가 없으면 **같은 쿠폰을 2번 사용**해 직접 손실이 발생한다. 화면 동작보다 **서버의 상태 전이(미사용→사용중→사용완료/롤백)** 정합성이 핵심이라 자동화·부하 테스트 가치가 높다. → TC-E02~E04

### R3. 적용 조건 검증의 서버단 일관성
명세는 "적용 불가 쿠폰은 비활성 표시"라고 화면 동작만 규정한다. **클라이언트 비활성화에만 의존하고 서버 검증이 없으면**, 요청 변조로 최소금액 미달·만료 쿠폰을 적용하는 악용이 가능하다. 금액과 직결되므로 보안 관점 검증이 반드시 필요하다. → TC-E05

## 4. 셀프 피드백 — 첫 작성본 대비 개선점

- **정확성 오류 수정:** "총 결제금액에서 차감" → "상품금액 합계 기준 차감, 배송비 제외"로 정정. (배송비 포함 여부는 S1으로 리포트)
- **TC ID 중복 제거**, 구분(정상/경계/예외)별로 ID 체계화(N/B/E).
- **시나리오 중복 작성 제거:** 별도 시나리오 문단을 없애고 사전조건·절차에 흡수 → 작성 시간 단축.
- **경계·예외 케이스 보강:** 첫 작성본은 정상 케이스만 있었음 → 경계 8개, 예외 6개 추가가 설계 역량의 핵심.
- **입력값·기대결과에 구체적 숫자** 기입, **우선순위 차등화**(High/Medium/Low).
