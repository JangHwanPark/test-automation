# T · Test Cases · 테스트 케이스 작성

주어진 기능 명세를 보고 테스트 케이스(TC)를 작성하는 카테고리.
실무에서는 Excel · Jira · TestRail 등으로 작성하지만, 본 과제집에서는 Markdown 표로 동일 형식을 따른다.

## 표준 양식

| TC ID | 대분류 | 소분류 | 사전 조건 | 테스트 단계 | 입력 데이터 | 기대 결과 | 우선순위 | 유형 | 비고 |
|---|---|---|---|---|---|---|---|---|---|
| TC-001 | 회원가입 | 정상 | 미가입 이메일 보유 | 1. 가입 화면 진입<br>2. 정상 정보 입력<br>3. 가입 버튼 클릭 | email: a@a.com<br>pw: Abc!1234<br>name: 홍길동 | 가입 성공 메시지 노출 + 자동 로그인 | P0 | Positive | - |

### 컬럼 정의
| 컬럼 | 설명 |
|---|---|
| TC ID | `TC-NNN` 순번 |
| 대분류 / 소분류 | 기능 트리 분류 (예: 회원가입 / 형식 오류) |
| 사전 조건 | 테스트 시작 전 갖춰져야 하는 상태 |
| 테스트 단계 | 1, 2, 3 … 순서대로 명확한 동작 |
| 입력 데이터 | 동등 분할 / 경계값 기반 |
| 기대 결과 | 단정 가능한 표현 (예: "○○ 메시지 노출", "버튼 disabled") |
| 우선순위 | P0(필수) / P1(중요) / P2(보통) / P3(낮음) |
| 유형 | Positive / Negative / Boundary / Negative-Boundary |
| 비고 | 의존 TC, 워크어라운드, 환경 특이 사항 |

## 작성 원칙
- **동등 분할(Equivalence Class)** 과 **경계값(Boundary Value)** 기법을 명시적으로 활용
- 한 TC = 한 검증 포인트 (한 단계에서 여러 단정 금지)
- 기대 결과는 "잘 동작한다" 같은 모호한 표현 금지
- 동일 입력의 반복은 데이터 테이블화로 압축 가능
- Positive : Negative : Boundary 비율은 대략 4 : 3 : 3 권장

## 평가 기준
| 기준 | 가중치 | 평가 내용 |
|---|---|---|
| 완전성 | 25% | 양식 컬럼 모두 채움, 케이스 누락 없음 |
| 기법 활용 | 25% | 동등 분할 / 경계값이 명시적으로 반영됨 |
| 우선순위 | 20% | P0~P3 분류 근거가 합리적 |
| 기대 결과 명확성 | 30% | 단정 가능, 모호 표현 없음 |
