# B · Bug Reports · 버그 리포트 작성

주어진 결함 상황을 표준 버그 리포트 양식으로 정리하는 카테고리.

## 표준 양식

| 항목 | 내용 |
|---|---|
| 제목 | `[모듈] 한 줄 요약` — 어떤 상황에서 어떤 문제가 발생하는지 |
| 보고자 | 이름 / 일시 |
| 환경 | OS, 브라우저 / 디바이스, 앱 버전, 빌드 번호, 네트워크 |
| 심각도 (Severity) | Critical / Major / Minor / Trivial |
| 우선순위 (Priority) | P0 / P1 / P2 / P3 |
| 사전 조건 | 재현에 필요한 사전 상태 |
| 재현 절차 | 1. ... <br>2. ... <br>3. ... |
| 기대 결과 | 정상 동작 시 기대되는 결과 |
| 실제 결과 | 현재 발생하는 결과 |
| 재현율 | 예: 5/5, 3/5, 1/10 |
| 첨부 | 스크린샷 / 영상 / 로그 / HAR / 콘솔 |
| 의심 원인 | (있다면) 어느 컴포넌트 / 어떤 변경이 의심되는지 |
| 워크어라운드 | (있다면) 임시 해결 방법 |
| 비고 | 관련 이슈, 영향 받는 사용자 범위 |

### Severity vs Priority
- **Severity** — 기능 자체에 미치는 영향 정도 (시스템 다운, 데이터 손실, 화면 깨짐 등)
- **Priority** — 비즈니스 관점에서 얼마나 빨리 고쳐야 하는지
- 둘은 독립적임 (예: Severity Critical / Priority P2, 또는 Severity Minor / Priority P0)

## 작성 원칙
- 제목은 **모듈 + 트리거 + 결과** 가 보이도록 (예: `[송금] 1,000만원 초과 송금 시 영상 인증 화면이 진입되지 않음`)
- 재현 절차는 **신규 사용자가 따라할 수 있을 정도**로 구체적으로
- 기대 결과 / 실제 결과는 **둘 다** 작성 (실제 결과만 적으면 안 됨)
- 추측은 의심 원인 항목에 분리, 절차 / 결과에는 사실만
- 재현이 불안정하다면 재현율을 반드시 표기

## 평가 기준
| 기준 | 가중치 | 평가 내용 |
|---|---|---|
| 재현성 | 30% | 절차만 보고도 재현 가능 |
| 분류 정확성 | 25% | Severity / Priority / 재현율 합리적 |
| 명확성 | 25% | 기대 / 실제 결과 분리, 모호 표현 없음 |
| 부가 정보 | 20% | 의심 원인 / 워크어라운드 / 환경이 풍부 |
